# 開工 / 收工 / 分支管理 / 合併 / 新專案初始化 / 架構生成

## #開工
1. 檢查當前資料夾：
   - 若資料夾為空或無專案檔案，主動要求使用者提供 GitHub 專案倉庫網址。
   - 若使用者同時提供了網址（例如：「#開工 https://github.com/hoonsor/...」）：
     - **檢查遠端分支**：在進行 `git clone` 之前，先利用 `git ls-remote --heads <URL>` 查詢該遠端倉庫的所有分支。
     - **分支確認與防呆 (重要)**：
       - 如果查詢結果顯示遠端除了預設主分支外，**還存在其他分支**，**請先暫停執行 clone**，並主動列出所有分支名稱向使用者確認：「*偵測到該遠端倉庫除了 main 之外，還有以下分支：[列出分支清單]。請問您本次是要使用 main 主分支作業，還是要在特定分支上作業？*」
       - 待使用者回答後，執行 `git clone <URL>`，並自動切換到使用者指定的對應分支（`git checkout <分支名稱>`）。
       - 如果遠端只有單一主分支，則直接執行 `git clone` 即可。
   - 若為新建立的空白專案（無 `ANTIGRAVITY.md`），自動套用「通用 ANTIGRAVITY.md 模板」建立初步架構。
2. 讀取 `ANTIGRAVITY.md`。若專案已成型但缺乏該檔案，應提示使用者使用 `#架構` 指令自動生成。
3. 讀取專案筆記重點（例如 `PROJECT_STATUS.md` 或專案筆記）。
4. **遠端同步檢查與自動拉取 (Git Sync Check)**：
   - 若當前專案已配置 Git，在背景執行 `git fetch origin`，查詢遠端倉庫是否有新的變動。
   - 檢查本地分支是否落後於遠端追蹤分支。
   - **自動 Pull 機制**：如果發現本地分支落後於遠端（顯示 `behind 'origin/xxx'`），**自動在背景執行 `git pull --rebase`**，將遠端最新變更安全地整併覆蓋至本地，確保開發起點是最新的。
   - 若在拉取過程中不幸發生衝突，立即暫停並向使用者回報。
5. 檢視當前的 `git status` 以及最近的本地/遠端 commit 紀錄。
6. 回報狀態（明確告知使用者是否進行了遠端程式碼更新）與建議下一步。

## #收工
1. 檢查敏感資料（API key、token、學生真名等）
2. 更新專案筆記（完成事項、下一步、踩坑）
3. 檢查專案修改情形，同步更新 `ANTIGRAVITY.md`（若技術棧、目錄結構或開發規則有變動）
4. 檢查 git status + diff
5. 只 stage 本次相關檔案（不用 `git add .`）
6. 確認後 commit + push
7. 回報同步結果

## #合併
當使用者輸入「#合併」指令時，依序執行以下 Git 合併與部署工作流：

1. **環境檢查與分支確認**：
   - 檢查當前資料夾是否處於 Git 管理中。若無 Git 設定，向使用者回報錯誤並停止。
   - 檢查當前所在的分支（Current Branch）。
   - **防呆與確認機制 (重要)**：
     - 若當前已處於 `main` 或是 `master` 主分支，無法自我合併。請主動發問向使用者確認：「*目前已處於 main 主分支，請問您是要將哪一個特定開發分支合併進來？*」
     - 若當前分支有未提交的變更（Uncommitted changes），請主動詢問使用者：「*偵測到當前分支有尚未提交的修改，請問要先執行 #收工 提交，還是進行暫存？*」
     - 如果有其他任何不確定或資訊模糊的地方，**請立刻停下來向使用者提問確認**，得到答案後再繼續。
2. **本地主分支更新**：
   - 暫存目前開發分支名稱（例如 `feature/minimax`）。
   - 切換至主分支：`git checkout main`。
   - 拉取遠端最新代碼，確保本地主分支為最新狀態：`git pull origin main`。
3. **執行合併與衝突處理**：
   - 將剛才的開發分支合併到主分支：`git merge <開發分支>`。
   - **衝突防範與處理 (重要)**：如果合併過程中發生代碼衝突（Conflicts），**請立刻暫停**，列出所有衝突的檔案並向使用者報告，詢問要如何處理衝突，不可擅自強行提交。
4. **推送與部署**：
   - 合併成功且無衝突後，將本地主分支推送到遠端倉庫：`git push origin main`。這通常會自動觸發 Vercel 等平台的自動部署。
5. **切回開發分支與回報**：
   - 執行完畢後，切換回原本的開發分支：`git checkout <開發分支>`。
   - 向使用者回報合併與推送結果，並提供 Vercel 部署更新確認。

## #架構
當專案已成型（已有程式碼檔案），但資料夾中缺少 `ANTIGRAVITY.md` 檔時，使用者輸入 `#架構` 指令，你必須：
1. 掃描整個專案資料夾的所有主要程式碼檔案與目錄結構。
2. 分析技術棧（主要語言、框架、相依套件檔如 package.json/requirements.txt 等）。
3. 根據分析結果，自動在專案根目錄下產製對應的 `ANTIGRAVITY.md` 檔案，填入該專案的實際技術細節與核心結構。

## #分支
當使用者輸入「#分支」指令時，請根據使用者提供的參數，執行以下兩種場景之一的作業：

### 宇宙 A：空白資料夾 + 遠端 Git 倉庫
* **前提條件**：使用者設定了一個「空的專案目錄資料夾」，並且貼上了「GitHub 倉庫網址」。
* **分支防呆與確認**：
  - 若使用者**沒有提供分支名稱**：**請先暫停執行**，主動詢問使用者：「*偵測到您提供了 GitHub 倉庫網址，請問您想為此專案建立並使用哪一個分支名稱？*」待使用者回覆並確認意圖後，再繼續執行。
  - 若已提供分支名稱，則直接執行以下步驟。
* **執行動作**：
  1. 將該 GitHub 倉庫 clone 到使用者設定的專案路徑中：`git clone <倉庫網址> <專案路徑>`。
  2. 進入該專案路徑，檢查該分支是否已在遠端或本地存在：
     - 若分支已存在：直接切換過去：`git checkout <分支名稱>`。
     - 若分支不存在：在本地新建分支並推送到遠端：`git checkout -b <分支名稱>`，隨後執行 `git push -u origin <分支名稱>` 在 GitHub 上建立對應的遠端分支並綁定。

### 宇宙 B：空白資料夾 + 本地 Git 專案路徑 (Worktree 模式)
* **前提條件**：使用者設定了一個「空的專案目錄資料夾」，並且貼上了「另一個本地專案路徑」。
* **分支防呆與確認**：
  - 若使用者**沒有提供分支名稱**：**請先暫停執行**，主動詢問使用者：「*偵測到您提供了本地專案路徑，請問您希望在新的專案目錄中檢出並使用哪一個分支名稱？*」待使用者回覆並確認意圖後，再繼續執行。
  - 若已提供分支名稱，則直接執行以下步驟。
* **本地 Git 環境檢查**：
  - 先檢查使用者提供的本地專案路徑中是否含有 Git 設定（檢查是否有 `.git` 資料夾，或在該路徑下執行 `git rev-parse --is-inside-work-tree`）。
  - 若沒有 Git 設定，向使用者回報錯誤並停止執行。
* **執行動作 (建立 Worktree)**：
  1. 切換到該「本地專案路徑」目錄下。
  2. 檢查該分支名稱是否已存在，若不存在，先建立它：`git branch <分支名稱>`。
  3. 執行 Git Worktree 指令，將該分支檢出到使用者指定的「空的專案目錄資料夾」中：
     `git worktree add <空的專案目錄資料夾路徑> <分支名稱>`
  4. 這會將該分支後續的所有開發作業，直接定位到您設定的那個專案目錄資料夾下。

## 新專案初始化
先問：名稱、用途、資料夾、是否 GitHub repo、公開/私有、是否部署。
建立：ANTIGRAVITY.md、README.md、.gitignore、Git repo、GitHub repo、專案筆記。
若已存在 → 盤點後只補缺口，不覆蓋。

    *   開發分支：<例如：請在 feature/ 相關分支上開發>

## ⚠️ 絕對禁止事項 (Never Do)
1.  **嚴禁提交敏感資訊**：嚴禁將 API Keys、Tokens、密碼、或個人隱私資訊寫死在程式碼中。
2.  **嚴禁直接破壞主分支**：未經測試與確認前，禁止對 main/master 分支進行強制推送 (--force)。
3.  **嚴禁覆蓋既有功能**：在修改代碼前，必須確認不會破壞已存在的正常功能，必要時先執行現有測試。
